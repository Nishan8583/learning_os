# 5.4 Boot Loaders - Notes

## What Is a Boot Loader?

A **boot loader** is the program that runs **before the Linux kernel**.

Its primary job is to:

1. Locate the Linux kernel on disk.
2. Load the kernel into memory.
3. Pass kernel parameters.
4. Transfer control to the kernel.

Basic flow:

```
Power On
    ↓
BIOS / UEFI
    ↓
Boot Loader
    ↓
Linux Kernel
    ↓
init / systemd
    ↓
User Space
```

---

# Why Is a Boot Loader Needed?

At first glance, booting Linux seems simple:

```
Load kernel
Start kernel
```

However, there is a major problem:

### Chicken-and-Egg Problem

To load the kernel, the boot loader must answer:

```
Where is the kernel?
What parameters should be passed?
```

Typically:

```
Kernel Image
Kernel Parameters
    ↓
Stored on Filesystem
```

But:

- The kernel is not running yet.
- Kernel filesystem drivers are unavailable.
- Kernel disk drivers are unavailable.

Therefore:

```
Need kernel
    ↓
To read filesystem
    ↓
But filesystem support is inside kernel
```

This is the bootstrapping problem.

---

# How Boot Loaders Access Disks

Boot loaders do **not** use Linux drivers.

Instead they use firmware services provided by:

### BIOS

**Basic Input/Output System**

Traditional PC firmware.

### UEFI

**Unified Extensible Firmware Interface**

Modern replacement for BIOS.

Boot flow:

```
Firmware
(BIOS/UEFI)
      ↓
Boot Loader
      ↓
Kernel
```

---

# Logical Block Addressing (LBA)

Modern firmware provides access to disks using:

```
LBA(Logical Block Addressing)
```

Instead of:

```
CylinderHeadSector
```

disks are viewed as:

```
Block 0
Block 1
Block 2
Block 3
...
```

Example:

```
Read Block 10000
```

rather than:

```
Cylinder 30
Head 2
Sector 17
```

---

## Why Use LBA?

Advantages:

- Universal interface
- Works with different storage devices
- Simple to implement

Disadvantage:

- Slow compared to native drivers

Not a problem because:

```
Boot Loader
    ↓
Uses LBA briefly
    ↓
Kernel starts
    ↓
Kernel loads optimized drivers
```

# BIOS vs UEFI

## Check Using efibootmgr

Command:

```
efibootmgr
```

### UEFI System

Example:

```
Boot0000
Boot0001
Boot0002
```

Meaning:

```
System uses UEFI
```

### BIOS System

Example:

```
EFI variables are not supported
```

Meaning:

```
System uses legacy BIOS
```

---

## Alternative Check

Command:

```
ls /sys/firmware/efi
```

or

```
test -d /sys/firmware/efi && echo UEFI
```

If directory exists:

```
UEFI system
```

If absent:

```
BIOS system
```

---

# Reading the Filesystem

Once raw disk access works:

```
Boot Loader
    ↓
Reads partition table
    ↓
Finds filesystem
    ↓
Reads kernel file
```

Modern boot loaders contain:

### Built-in filesystem support

Examples:

- ext4
- FAT
- XFS
- Btrfs (varies by loader)

This allows:

```
/boot/vmlinuz/boot/initrd.img
```

to be loaded directly.

---

# Historical Boot Loaders

Older boot loaders often lacked filesystem support.

Instead:

```
Kernel blockswere manually specified
```

Result:

- Harder configuration
- More maintenance
- Less flexibility

Modern boot loaders are much easier to manage.

---

# Evolution of Boot Loaders

A common pattern in Linux:

```
Kernel gains new storage feature
          ↓
Boot loader implements
a simplified version
```

Examples:

```
Partition support
Filesystem support
LVM support
RAID support
UEFI support
```

This allows the boot loader to locate the kernel before Linux starts.

# 5.4.1 Boot Loader Tasks

A Linux boot loader typically supports:

---

## 1. Multiple Kernels

Example:

```
Partition support
Filesystem support
LVM support
RAID support
UEFI support
```

Useful when:

- New kernel fails
- Testing custom kernels

Flow:

```
Boot Menu
    ↓
Choose Kernel
```

---

## 2. Different Kernel Parameters

Example:

Normal boot:

```
quiet splash
```

Debug boot:

```
debug loglevel=7
```

Recovery boot:

```
single
```

Boot loaders can store multiple parameter sets.

---

## 3. Manual Editing

At boot time you can edit:

```
Kernel Image
Kernel Parameters
```

Example:

```
linux /boot/vmlinuz root=/dev/sda1 single
```

Common uses:

- Recovery
- Troubleshooting
- Debugging

---

## 4. Boot Other Operating Systems

Examples:

```
Linux
Windows
FreeBSD
```

Example menu:

```
GRUB Menu

Linux
Windows 11
Memtest
```

---

# Why Manual Booting Matters Less Today

Historically:

```
Broken system
    ↓
Single-user mode
    ↓
Repair machine
```

Today:

```
Broken system
    ↓
Boot Live USB
    ↓
Repair machine
```

Therefore manual kernel parameter entry is less frequently needed.

# 5.4.2 Boot Loader Overview

## GRUB

Full name:

```
GRand Unified Bootloader
```

Most common Linux boot loader.

Supports:

- BIOS
- UEFI
- Multiple OSes
- Multiple filesystems
- Recovery options

Typical path:

```
/boot/grub/
```

Example menu:

```
Ubuntu
Advanced Options
Windows
```

---

## LILO

**Linux Loader**

One of the earliest Linux boot loaders.

Characteristics:

- Simple
- Predates GRUB
- Largely obsolete

UEFI version:

```
ELILO
```

---

## SYSLINUX

Can boot from many media types:

- FAT filesystems
- USB drives
- CDs/DVDs

Common in:

```
Rescue media
Live distributions
```

---

## LOADLIN

Special boot loader that:

```
MS-DOS
   ↓
Linux Kernel
```

Boots Linux from DOS.

Mostly historical today.

---

## systemd-boot

Simple UEFI boot manager.

Characteristics:

- Small
- Fast
- Easy configuration

Works only on:

```
UEFI systems
```

---

## coreboot

Formerly:

```
LinuxBIOS
```

Not just a boot loader.

It replaces the BIOS itself.

Goals:

- Faster startup
- Open-source firmware
- Custom hardware support

Flow:

```
Power On
    ↓
coreboot
    ↓
Kernel
```

---

## Linux Kernel EFISTUB

Allows:

```
UEFI
   ↓
Kernel Directly
```

without GRUB.

Kernel behaves like a UEFI application.

Flow:

```
UEFI
   ↓
vmlinuz
   ↓
Linux
```

---

## efilinux

A reference UEFI boot loader.

Primarily useful for:

- Development
- Learning
- Testing

Less common on production systems.

---

# Why GRUB Dominates Linux

GRUB supports:

✅ BIOS

✅ UEFI

✅ Multiple operating systems

✅ Filesystems

✅ Recovery options

✅ Kernel editing

Because of this, most Linux distributions use GRUB.

# Complete Boot Sequence

```
Power On
    ↓
BIOS / UEFI
    ↓
Boot Loader (GRUB)
    ↓
Read partition table
    ↓
Read filesystem
    ↓
Load kernel
    ↓
Load initramfs
    ↓
Pass kernel parameters
    ↓
Transfer control
    ↓
Linux Kernel
    ↓
init/systemd
    ↓
User Space
```
