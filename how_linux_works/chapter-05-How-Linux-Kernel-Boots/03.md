# 5.5 GRUB (Grand Unified Boot Loader) - Notes
One of GRUB's most important features is its ability to:

- Navigate filesystems
- Locate kernel images
- Locate initramfs images
- Load GRUB modules dynamically
- Support multiple operating systems

- Hold `SHIFT` (BIOS) or `ESC` (UEFI) to access GRUB menu.
- In Grub menu `e` to access configuration editor.
- `c` To open GRUB shell.
- NOTE: Its a different shell from linux, even though command names are similar, its a whole different world and its just that naming pattern are same.
---
# Example GRUB Boot Entry

A GRUB boot entry contains:
```
set root='hd0,msdos1'

search --fs-uuid ...

linux /boot/vmlinuz-4.15.0-45-generic \
      root=UUID=... \
      ro quiet splash

initrd /boot/initrd.img-4.15.0-45-generic
```
- There are actually **two different roots**.
- `Linux Root Filesystem` passe to the kernel:
```
linux /boot/vmlinuz \
root=UUID=...
```
- which becomes `/` after linux boots.
- `GRUB Root` only inside GRUB.
```
set root=(hd0,msdos1)
```
- This tells grub to look for kernel files here.
- GRUB root and Linux root are completely different concepts.

# MBR Boot Process
- Master Boot Record (a sector on disk).
- Does not refer to the partition here.
- It is the first sector of the disk (LBA 0), traditionally 512 bytes:
```
Disk
├── MBR sector
│   ├── Boot code
│   └── Partition table
├── Partition 1
├── Partition 2
└── ...
```
- A traditional MBR layout looks like:
```
Offset      Size
--------------------------------
Boot code   446 bytes
Partitions   64 bytes
Signature     2 bytes
--------------------------------
Total       512 bytes
```
Traditional BIOS systems:

```
Power On
   ↓
BIOS
   ↓
MBR (441 bytes)
   ↓
GRUB Stage 1
   ↓
GRUB Stage 2
   ↓
Kernel
```
- So "MBR Boot" does not mean:

`Boot from an MBR partition`

because there is no "MBR partition".

It means:

`Boot using BIOS and the Master Boot Record boot mechanism.`
---

## MBR Limitation

MBR contains only:

```
441 bytes
```

of executable code.

Too small for GRUB.

Therefore:

```
Stage 1
    ↓
Loads Stage 2
```

stored elsewhere on disk.
- AKA `Multi Stage Boot Loader`.

Special FAT filesystem.

Usually mounted at:

```

```


## BIOS

**BIOS = Basic Input/Output System**

This is the traditional PC firmware from the 1980s.

When you power on:

```
CPU starts
   ↓
BIOS runs
   ↓
Checks RAM, keyboard, disks
(POST)
   ↓
Reads first sector of boot disk
(MBR)
   ↓
Executes boot code
```

The BIOS doesn't understand:

- ext4
- FAT
- files
- directories

It only knows:

```
Read sector N from disk
```

That's why boot loaders on BIOS systems need the complicated multi-stage setup.

---

## UEFI

**UEFI = Unified Extensible Firmware Interface**

UEFI is the modern replacement for BIOS.

Instead of loading a tiny 440-byte boot sector:

```
UEFI
   ↓
Reads FAT filesystem
   ↓
Finds .efi file
   ↓
Runs it
```

Example:

```
/boot/efi/EFI/ubuntu/grubx64.efi
```

UEFI understands:

- GPT partition tables
- FAT filesystems
- Files
- Directories
- NVRAM boot entries

So booting is much cleaner.

# 5.8.3 How GRUB Works: 
1. The PC BIOS or firmware initializes the hardware and searches its boot-order storage devices for boot code. 
2. Upon finding the boot code, the BIOS/firmware loads and executes it. This is where GRUB begins. 
3. The GRUB core loads. 
4. The core initializes. At this point, GRUB can now access disks and filesystems. 
5. GRUB identifies its boot partition and loads a configuration there. 
6. GRUB gives the user a chance to change the configuration. 
7. After a timeout or user action, GRUB executes the configuration (the sequence of commands in the grub.cfg file, as outlined in Section 5.5.2). 
8. In the course of executing the configuration, GRUB may load additional code (modules) in the boot partition. Some of these modules may be preloaded.
9. GRUB executes a boot command to load and execute the kernel as specified by the configuration’s linux command. 

Steps 3 and 4 of this sequence, where the GRUB core loads, can be complicated due to the inadequacies of traditional PC boot mechanisms. The biggest question is “Where is the GRUB core?” There are three basic possibilities: 
- Partially stuffed between the MBR and the beginning of the first partition 
- In a regular partition
- In a special boot partition: a GPT boot partition, ESP, or elsewhere

In all cases except where you have an UEFI/ESP, the PC BIOS loads 512 bytes from the MBR, and that’s where GRUB starts. This little piece (derived from boot.img in the GRUB directory) isn’t yet the core, but it contains the start location of the core and loads the core from this point.

---

### The MBR sector is 512 bytes

Traditionally, the first sector on a disk is:

```
512 bytes
```

Layout:

```
+----------------------+
| Boot code            | ~446 bytes
+----------------------+
| Partition table      | 64 bytes
+----------------------+
| Signature (0x55AA)   | 2 bytes
+----------------------+
Total = 512 bytes
```

So when the book says:

> "The PC BIOS loads 512 bytes from the MBR"

that's literally true.

BIOS reads the **entire first sector** into memory and jumps to it.

---

### Why did the earlier section say 441 bytes?

Because not all 512 bytes are available for bootloader code.

The boot code area is approximately:

```
446 bytes
```

but some bytes are often reserved for:

- Disk signature
- Miscellaneous metadata

Leaving roughly:

```
440-446 bytes
```

available for executable code.

Different authors quote:

- 440 bytes
- 441 bytes
- 446 bytes

depending on exactly what they count.

---

### What BIOS Actually Does

```
BIOS
   ↓
Read 512-byte MBR sector
   ↓
Load it into memory
   ↓
Jump to first instruction
```

But GRUB only gets about:

```
~440 bytes
```

of actual executable space in that sector.

---

### Analogy

Think of a 512-byte envelope:

```
Envelope size = 512 bytes
```

Inside:

```
446 bytes  -> Boot code
64 bytes   -> Partition table
2 bytes    -> Magic signature
```

The BIOS delivers the entire envelope.

But GRUB can only write code in the boot-code portion.

---

### So Both Statements Are Correct

When the book says:

> BIOS loads 512 bytes from the MBR

it means:

```
Entire sector is loaded.
```

When it says:

> The MBR includes a small area of 441 bytes

it means:

```
Only ~441 bytes are availablefor bootloader code.
```

That's exactly why GRUB needs multiple stages:

```
512-byte MBR sector loaded
        ↓
~440 bytes of Stage 1 code
        ↓
Loads GRUB core
        ↓
Loads full GRUB
        ↓
Loads Linux
```

So there is no contradiction:

- **512 bytes** = size of the entire MBR sector read by BIOS.
- **~441 bytes** = usable bootloader code space within that sector.
