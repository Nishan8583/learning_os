# 5. How Linux Kernel Boots

A simplified view of the boot process looks like this: 
1. The machine’s BIOS or boot firmware loads and runs a boot loader. 
2. The boot loader finds the kernel image on disk, loads it into memory, and starts it. 
3. The kernel initializes the devices and its drivers.
4. The kernel mounts the root filesystem. 
5. The kernel starts a program called init with a process ID of 1. This point is the user space start. 
6. init sets the rest of the system processes in motion. 
7. At some point, init starts a process allowing you to log in, usually at the end or near the end of the boot sequence.

### 5.1 Startup Messages
- Messages come from kernel first.
- Then from Processes initalized by `init`.
- Too fast, modern OSes show splash screen.
- `journalctl -k` see messages from current boot.
- `journalctl -b` see messages from previous boot as well.
- If no systemd `/var/log/kern.log` or run the `dmesg` command
### 5.2 Kernel Initialization and Boot Options

Upon startup, the Linux kernel initializes in this general order: 
1. CPU inspection 
2. Memory inspection 
3. Device bus discovery 
4. Device discovery 
5. Auxiliary kernel subsystem setup (networking and the like)
6. Root filesystem mount 
7. User space start

Some machines may need to load these kernel modules before the true root filesystem is mounted

- The kernel emits certain kinds of messages indicating that it’s getting ready to start its first user process:
```
Freeing unused kernel memory: 2408K 
Write protecting the kernel read-only data: 20480k 
Freeing unused kernel memory: 2008K 
Freeing unused kernel memory: 1892K
```

- Then, if you’re running a new enough kernel, you’ll see the kernel start the first user-space process as init: 
```
  Run /init as init process 
	  with arguments: --snip--
```

- Root file system being mounted and systemd startup messages
```
  EXT4-fs (sda1): mounted filesystem with ordered data mode. Opts: (null) systemd[1]: systemd 237 running in system mode. (+PAM +AUDIT +SELINUX +IMA +APPARMOR +SMACK +SYSVINIT +UTMP +LIBCRYPTSETUP +GCRYPT +GNUTLS +ACL +XZ +LZ4 +SECCOMP +BLKID +ELFUTILS +KMOD -IDN2 +IDN -PCRE2 default hierarchy=hybrid)
```

# 5.3 Kernel Parameters - Notes

## What Are Kernel Parameters?

When the Linux kernel boots, it receives a set of **text-based kernel parameters** that control kernel behavior and provide important system information.

These parameters can:

- Control kernel diagnostic output
- Configure device drivers
- Specify the root filesystem
- Control boot modes
- Pass options to user-space initialization programs

---

## Viewing Current Kernel Parameters

The currently running kernel's boot parameters can be viewed through:

```
cat /proc/cmdline
```

Example output:

```
$ cat /proc/cmdline

BOOT_IMAGE=/boot/vmlinuz-4.15.0-43-generic \
root=UUID=17f12d53-c3d7-4ab3-943e-a0a72366c9fa \
ro quiet splash vt.handoff=1
```

### `/proc/cmdline`

A virtual file created by the kernel that contains the exact command line used to boot the current kernel.

---

## Types of Kernel Parameters

### 1. Simple Flags

Single-word parameters:

```
ro
quiet
splash
```

Examples:

|Parameter|Purpose|
|---|---|
|`ro`|Mount root filesystem read-only initially|
|`quiet`|Reduce boot messages|
|`splash`|Display graphical splash screen|

---

### 2. Key=Value Parameters

Parameters with explicit values:

```
vt.handoff=1root=UUID=17f12d53-c3d7-4ab3-943e-a0a72366c9fa
```

General format:

```
parameter=value
```

---

# The Important `root=` Parameter

The most critical boot parameter is:

```
root=
```

It tells the kernel:

> "Where is the root filesystem?"

Without it:

- Kernel cannot locate `/`
- User-space startup cannot continue
- System boot fails

---

## Root Filesystem Specification Methods

### 1. Traditional Device File

Example:

```
root=/dev/sda1
```

Meaning:

```
/dev/sda1
 └─ First partition on first SATA/SCSI disk
```

### Problem

Device naming can change:

```
Today:
/dev/sda = SSD

Tomorrow:
/dev/sda = USB drive
```

Result:

```
Wrong root filesystem mounted
```

---

### 2. Logical Volume (LVM)

Example:

```
root=/dev/mapper/my-system-root
```

Used when:

- System uses LVM
- Root filesystem resides inside a logical volume

Flow:

```
Physical Disk
      ↓
Volume Group
      ↓
Logical Volume
      ↓
Filesystem
```

Advantages:

- Flexible storage management
- Independent of physical disk names

---

### 3. UUID (Most Common)

Example:

```
root=UUID=17f12d53-c3d7-4ab3-943e-a0a72366c9fa
```

UUID = Universally Unique Identifier

Example:

```
17f12d53-c3d7-4ab3-943e-a0a72366c9fa
```

Advantages:

- Unique for each filesystem
- Doesn't depend on device names
- Survives disk reordering

Preferred on modern Linux systems.

---

# Why UUIDs and LVM Paths Are Better

Instead of:

```
root=/dev/sda1
```

Use:

```
root=UUID=...
```

or

```
root=/dev/mapper/...
```

Because:

```
Device names can change
UUIDs do not
```

Examples:

```
/dev/sda1 → /dev/sdb1
```

after:

- adding disks
- changing BIOS settings
- plugging in USB devices

UUID remains constant.

---

# The `ro` Parameter

Example:

```
ro
```

Meaning:

```
Mount root filesystem as Read-Only
```

during early boot.

---

## Why Mount Read-Only First?

Before normal operation, Linux may need to run:

```
fsck
```

(File System Check)

Running fsck on a writable filesystem is dangerous.

Read-only mode ensures:

- Consistent filesystem state
- Safe repairs
- No writes during checking

---

## Boot Flow with `ro`

```
Kernel boots
      ↓
Mount root filesystem (read-only)
      ↓
Run fsck
      ↓
Filesystem verified/repaired
      ↓
Remount root filesystem (read-write)
      ↓
Continue boot
```

---

# Unknown Kernel Parameters

The kernel does not necessarily reject unknown parameters.

When it encounters one it doesn't understand:

```
Kernel
   ↓
Stores parameter
   ↓
Passes it to init
```

This allows parameters intended for user-space programs.

---

## Example: Single User Mode

Boot parameter:

```
-s
```

Kernel behavior:

```
Kernel does not use it
      ↓
Passes it to init
      ↓
init starts single-user mode
```

---

### Single User Mode

A maintenance/recovery mode:

- Minimal services
- Usually root access
- Useful for repairs

Conceptually:

```
Kernel
   ↓
init -s
   ↓
Recovery Shell
```

# Common Example Kernel Command Line

```
BOOT_IMAGE=/boot/vmlinuz-4.15.0-43-generic
root=UUID=17f12d53-c3d7-4ab3-943e-a0a72366c9fa
ro
quiet
splash
vt.handoff=1
```

Breakdown:

|Parameter|Meaning|
|---|---|
|`BOOT_IMAGE=`|Kernel image used to boot|
|`root=`|Root filesystem location|
|`ro`|Mount root filesystem read-only|
|`quiet`|Reduce boot messages|
|`splash`|Show splash screen|
|`vt.handoff=1`|Virtual terminal handoff setting|

---

# Where Do Kernel Parameters Come From?

Typically supplied by the bootloader:

```
BIOS/UEFI
    ↓
GRUB
    ↓
Linux Kernel
```

Example GRUB entry:

```
linux /boot/vmlinuz root=UUID=... ro quiet splash
```

The bootloader appends these parameters when loading the kernel.
